# Contractility

Локальный прототип для распознавания сканированных договоров и подготовки
дополнительных соглашений. PDF.js отрисовывает страницы в браузере,
Tesseract.js распознаёт русский и английский текст, а результат связывается с
отдельно выбранной новой редакцией допсоглашения DOCX через контрольные суммы.

## Запуск в закрытом контуре

На целевом компьютере `npm install` не выполняется. Репозиторий уже содержит
необходимые браузерные файлы, WebAssembly-ядро и языковые модели в
`public/vendor`.

Требования времени выполнения:

- Node.js 22 или новее;
- актуальный Chrome, Edge или Firefox с WebAssembly и Web Workers.

```bash
node scripts/doctor.mjs
node src/server.mjs
```

После запуска откройте `http://127.0.0.1:4317`.

Сервер использует только стандартную библиотеку Node.js, слушает loopback-адрес
и не предоставляет API загрузки документов. PDF выбирается непосредственно в
браузере и не отправляется во внешнюю сеть.

## Возможности текущего MVP

- открытие комплекта локальных PDF без загрузки на сервер: первый файл
  определяется как исходный подписанный договор, остальные — как подписанные
  дополнительные соглашения;
- изменение порядка подписанных допсоглашений без возможности переместить их
  перед исходным договором;
- добавление новых PDF в уже выбранный комплект без сброса обработанных страниц;
- отдельная загрузка новой редакции дополнительного соглашения в DOCX: она не
  попадает в цепочку подписанных документов;
- использование готового текстового слоя, если он присутствует;
- OCR сканированных страниц на русском и английском языках;
- настройка 160, 220 или 300 DPI;
- автоматическое приведение альбомных сканов к книжной ориентации и ручной
  выбор поворота на 90°/180° для нестандартных документов;
- текст, уверенность OCR и координаты строк по каждой странице;
- ручное исправление распознанного текста;
- экспорт `*.formation-request.json` и `*.formation-request.txt` со строгим
  четырёхэтапным заданием на реконструкцию и формирование;
- SHA-256 каждого PDF, новой редакции DOCX и vendored-компонентов;
- полностью локальные модели без CDN.

```bash

```

## Диагностика на целевом компьютере

Если страницы завершаются с ошибкой внутри браузерного OCR, сначала выполните
обычные проверки, затем запустите диагностическую страницу:

```bash
npm test
node scripts/doctor.mjs
npm run diagnostics
```

В Safari откройте `http://127.0.0.1:4317/diagnostics.html`, выберите проблемный
PDF, дождитесь завершения и скачайте `*.diagnostics.json`. Проверка обрабатывает
только первую страницу на 220 DPI, повторяет точный вызов приложения, проверяет
структуру OCR-блоков и постобработку строк. При ошибке распознавания она также
проверяет передачу изображения через Data URL. Отчёт содержит версию Safari,
этапы и стеки ошибок, но не содержит изображение или распознанный текст.
Если Safari падает уже на `PDF.js getTextContent`, проверка завершается сразу:
основное приложение должно пропустить текстовый слой и перейти к обычному OCR.

## Обновление vendored-компонентов

Команды этого раздела выполняются только на контролируемой машине сборки с
доступом к npm, а не на целевом компьютере:

```bash
npm install --ignore-scripts
npm run vendor
npm test
```

`npm run vendor` копирует только требуемые runtime-файлы и создаёт
`public/vendor/manifest.json` с SHA-256 каждого файла. Версии зафиксированы в
`package.json` и `package-lock.json`. Сведения о лицензиях приведены в
`THIRD_PARTY_NOTICES.md`.

## Формат результата

Внутренний OCR-результат использует схему `contractility.ocr.v2`. Итоговый
экспорт использует схему `contractility.formation-request.v1` и содержит:

- упорядоченный массив подписанных документов с ролями `contract` и
  `additional-agreement`;
- метаданные и SHA-256 отдельно передаваемой новой редакции DOCX;
- четыре обязательных операции: реконструкция исходной редакции,
  последовательное применение подписанных изменений, сравнение с новой
  редакцией и генерация финального допсоглашения;
- правила разрешения коллизий, сохранения структуры DOCX, привязки каждого
  изменения к источнику и обязательного ручного подтверждения.

Для каждой распознанной страницы дополнительно хранятся:

- источник текста (`pdf-text` или `tesseract`);
- текст и уверенность;
- время обработки;
- поворот страницы;
- размеры изображения, переданного OCR;
- нормализованные координаты распознанных строк;
- признак ручного исправления.

DOCX не встраивается в JSON: он передаётся семантическому модулю отдельным
файлом, а совпадение проверяется по SHA-256. Это позволяет сохранить таблицы,
сноски, поля, стили и связи исходного OOXML-пакета.

Target CLI исполняет `contractility.formation-request.v1`, сохраняет реестр
применённых изменений, проводит циклическое межмодельное ревью и выпускает
финальный DOCX только после ручного подтверждения проверенного хеша.

## Запуск семантического контура на целевой машине

В репозитории есть отдельный target CLI, рассчитанный на Node.js 22+, GigaCode
CLI, `zip`/`unzip` и, для обязательного визуального прогона, LibreOffice
`soffice`. На целевой машине не выполняется `npm install`: все runtime-файлы
приложения уже находятся в комплекте.

1. Скопируйте конфигурацию и укажите реальные идентификаторы доступных моделей:

```bash
cp config/target.example.json config/target.json
```

Минимум три разных модели обязательны. Отдельные модели задаются для producer,
арбитра и каждого reviewer. Проверить CLI, зависимости и все модели:

```bash
npm run doctor:target -- --config config/target.json --smoke
```

2. Подготовьте неизменяемый case bundle. Идентификаторы в `--source` должны
совпадать с `inputs.signedDocuments[].id` из formation request:

```bash
npm run target -- prepare \
  --request incoming/contract.formation-request.json \
  --draft incoming/new-edition.docx \
  --source document-1=incoming/contract.pdf \
  --source document-2=incoming/agreement-1.pdf \
  --out data/cases
```

Команда повторно считает SHA-256 каждого PDF и DOCX, проверяет порядок,
копирует входы в приватный каталог и создаёт `case-manifest.json`. Несовпадение
любого файла останавливает процесс.

3. Запустите формирование и циклическое ревью:

```bash
npm run target -- run \
  --case data/cases/CASE_ID \
  --config config/target.json
```

Пайплайн:

1. producer реконструирует действующую редакцию и создаёт первый кандидат;
2. детерминированные проверки контролируют DOCX и защищённые OOXML-части;
3. независимые reviewer-агенты на разных моделях запускаются параллельно;
4. единственный synthesis-арбитр перепроверяет все замечания по источникам;
5. подтверждённые дефекты исправляются, после чего весь review повторяется;
6. повторяющийся кандидат и одинаковый набор замечаний переводят запуск в
   `blocked`, а не в ложный успех;
7. успешный автоматический этап останавливается в
   `awaiting-human-approval`.

Роли по умолчанию: реконструкция договора, юридическая дельта, внутренние
ссылки и реквизиты, целостность DOCX, доказательность и безопасность.

4. Проверьте состояние и подтвердите ровно тот кандидат, который прошёл review:

```bash
npm run target -- status --run data/runs/RUN_ID

npm run target -- approve \
  --run data/runs/RUN_ID \
  --candidate-sha256 HASH_ИЗ_STATE \
  --findings-sha256 HASH_ИЗ_STATE \
  --approver "ФИО проверяющего"

npm run target -- finalize --run data/runs/RUN_ID
npm run target -- verify --run data/runs/RUN_ID
```

`approval.json` является аудиторским подтверждением процесса, а не электронной
подписью. Финализация без него или после изменения кандидата невозможна.

## Параллельное ревью

Контур использует проверенный подход
[`RomanovAV/gigalphex`](https://github.com/RomanovAV/gigalphex):
параллельные read-only рецензенты, строгий формат замечаний, недоверенная
граница для reviewer output, единый арбитр и повторные review-раунды. В
Contractility этот подход расширен индивидуальной моделью для каждого
рецензента, юридическими ссылками на документ/страницу/пункт, проверкой хешей
входов после каждого агентного этапа и остановкой при отсутствии прогресса.

## Сборка переносимого комплекта

```bash
npm run bundle:target
```

Команда создаёт `dist/contractility-target.zip` и манифест с SHA-256. В архив
не попадают `.git`, `node_modules`, исходные документы, cases, runs и временные
файлы.

Семантическая стадия может обращаться к настроенному внутреннему сервису
моделей GigaCode. Поэтому формулировка «без отправки файлов в сеть» относится
только к браузерному OCR; допустимость передачи текста договоров в модельный
контур должна быть подтверждена политикой целевой организации.
